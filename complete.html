Centralized Food Ordering Portal: Virginia Tech

Introduction


Objective

This test plan's objective is to create a benchmark for all future testing of this system. In addition to providing guidance on the overall testing strategy, including the types of testing to be conducted, various testing phases, exit criteria, and so on, this plan will serve as the basis for the minimal testing that must be carried out.

The baseline for all efforts connected to testing should be this document, which should be an addition to the system requirements and architectural changes.


System description

The system is designed to make the Southgate Dining hall operate more efficiently. It will automate the Virginia Tech Dining Halls' centralized meal ordering system. Southgate would serve as the portal's supreme administrator. The Manager would update the list of food items that needed to be ordered from the vendor on a regular basis. The Food Ordering Portal will have separate accounts for each dining hall. Each dining hall's manager is in charge of submitting requests through the portal for the food item and both the required and desired quantities. The system stops accepting requests for the day at 1 PM and automatically contacts the appropriate vendors to request the food items. The system notifies the dining halls of the anticipated items for the following day and the best vendor responds with the item that is currently in stock. The portal reflects the delivery to each dining hall once the items have been delivered.


Scope

The scope of this test plan is only for integration and validation testing. We have outlined the specifications, design elements, and test strategy in great detail. For each special test method, we have a separate phase in our test plans that includes a brief explanation of why building all test case skeletons for the test cases is necessary. All of our test cases are built on top of the traceability matrix.


System Architecture

Looking at the functional and nonfunctional requirements, it was evident that there would be 3 parts of the system:
  1. Front-end or UI of the application
  2. Back-end logic for providing the required functionality of the portal
  3. Database support to store all the data in a coherent way and to make data manipulation more efficient
The high level structure which would be used to design this web application will be Model-View-Template structure.

Exit Criteria

We have the following exit criteria:
  • All planned testing has been completed.
  • There shouldn't be any unresolved high severity or critical faults.
  • All of the high-risk areas have been carefully examined.
  • The code's coverage requirements ought to be met.

Test Plan and Integration


The proposed solution is designed using Model-View-Template architecture. Model is basically the database part of the web application. View can be considered as the back-end of the application where all the core code resides. View also connects Model as well as Templates. Templates are HTML files which will be rendered on the front-end of the application. All the user interaction will happen with the front-end of the application. Similarly, all the required results will be displayed on the front-end.  

Testing strategy of the web application should test all 3 components of the architecture. If we can break down the testing plan for each component separately, it will further help us to narrow down the problem or bugs. Thus, bug fixing could be done more efficiently and accurately. Dividing the testing into different phases will further aid us to automate the testing phase using different tools such as Selenium, visual studio etc. Use of tools will further solidify the system components in terms of bug fixes.

Once the component testing is complete, we can move towards the system testing where we can check the behavior of the system, in different scenarios to make sure that the designed components are working properly and in cohesion with each other. In other words, we can say that the component testing is low level testing and system testing is high level testing. Therefore, it can be concluded that we are moving from lowest level to highest level. If there are certain issues in system testing, but not in component level, we have to check the compatibility of different components with each other. The different components will be tested for different edge cases and properties. They are as follows: -

  •  Front-end: Testing on the front-end will mostly be focused on the GUI of the application. Validations will be performed on different fields of HTML forms. E.g. fields should not accept some random text but should accept text related to orders only, Fields should not be kept empty etc.

  •  Back-end: Testing on back-end will be focused mostly on the core logic. Are we able to generate accurate results, are we able to retrieve the data from the front-end and are computed results successfully sent out to the front-end or not, along with the database connections will be checked. Edge cases will cover the core logic scenarios. API’s will be checked. User Authentication is also one of the key areas which will be checked during this phase.

  •  Database: Testing in this phase will be focused on the database queries, and database schema. We will also check if the database is following all the ACID properties. Edge cases will cover SQL injection attacks, and other security concerns related with the stored data.

Validation Tests

Test Procedure

Once development is complete, we will set up a timetable for validating testing to guarantee that we are checking out the final, deliverable portal. Validation testing will be used to ensure that all of the system's components are communicating effectively with one another. As soon as we get access to all of the necessary resources, we will experiment with a few things that we couldn't try out beforehand.

In addition to functionality, the product will be evaluated based on non-functional factors such as performance, load-bearing qualities, and safety. The following types of testing will be utilized in the course of our validation checks:
  • Functional Testing
  • Load Testing
  • Performance Testing
