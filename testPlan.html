Test Plan and Integration


The proposed solution is designed using Model-View-Template architecture. Model is basically the database part of the web application. View can be considered as the back-end of the application where all the core code resides. View also connects Model as well as Templates. Templates are HTML files which will be rendered on the front-end of the application. All the user interaction will happen with the front-end of the application. Similarly, all the required results will be displayed on the front-end.  

Testing strategy of the web application should test all 3 components of the architecture. If we can break down the testing plan for each component separately, it will further help us to narrow down the problem or bugs. Thus, bug fixing could be done more efficiently and accurately. Dividing the testing into different phases will further aid us to automate the testing phase using different tools such as Selenium, visual studio etc. Use of tools will further solidify the system components in terms of bug fixes.

Once the component testing is complete, we can move towards the system testing where we can check the behavior of the system, in different scenarios to make sure that the designed components are working properly and in cohesion with each other. In other words, we can say that the component testing is low level testing and system testing is high level testing. Therefore, it can be concluded that we are moving from lowest level to highest level. If there are certain issues in system testing, but not in component level, we have to check the compatibility of different components with each other. The different components will be tested for different edge cases and properties. They are as follows: -

  •  Front-end: Testing on the front-end will mostly be focused on the GUI of the application. Validations will be performed on different fields of HTML forms. E.g. fields should not accept some random text but should accept text related to orders only, Fields should not be kept empty etc.

  •  Back-end: Testing on back-end will be focused mostly on the core logic. Are we able to generate accurate results, are we able to retrieve the data from the front-end and are computed results successfully sent out to the front-end or not, along with the database connections will be checked. Edge cases will cover the core logic scenarios. APIâ€™s will be checked. User Authentication is also one of the key areas which will be checked during this phase.

  •  Database: Testing in this phase will be focused on the database queries, and database schema. We will also check if the database is following all the ACID properties. Edge cases will cover SQL injection attacks, and other security concerns related with the stored data.


Front-end Testing Phase

Testing will be focused on the system accessibility, system GUI and esthetic features along with looks of the GUI. The factors which will be tested are Design, functionality, performance.

Test Test Case No Test Case Description Passing Criteria
TIC-1 Compulsory Field validation Error messages should be displayed if any field is left blank. The compulsory fields should be marked with * or any other symbol.
TIC-2 Form Validation Form should be correctly populated and submitted. Validations should be implemented for every field to make sure only acceptable text is accepted. If not, Error message should be displayed.
TIC-3 CSS and visual effects validation CSS and all other visual esthetic contents should be loaded without any errors. If not, the web page should not be loaded and appropriate error messages should be displayed.
TIC-4 Dynamic Content Validation Any dynamic content which will be displayed should be correctly loaded.

Back-end will have the core business logic of the web application. Therefore, back-end testing will be the most important phase. We have to test the business logic along with connections and data-flow from back-end to front-end and back-end to database. System security should also be tested. APIâ€™s calls should be giving expected results. HTTP headers should be checked. This phase will consist of multiple sub-testing phases to ensure we are not missing out on anything. Database Testing Phase

In this phase we will test the Database connections, Database ACID properties and data Integrity. Along with this, some security checks should also be tested out. For e.g. SQL injection attack, Data Leak check etc. Database Schema will also be tested.


Test Test Case No Test Case Description Passing Criteria
TIC-14 Database schema testing The designed database schema should be in-line with the expected design database schema.
TIC-15 SQL procedure testing The stored SQL procedures should give the expected results. If not, the procedure should be able to handle the incorrect outputs.
TIC-16 Database Definition Testing The designed ER diagrams should be followed.
TIC-17 Security Testing SQL injection attacks, Data Leak Attacks etc. should be checked and the system should be able to handle all the attacks.
TIC-18 Important Data Encryption Testing The stored user password should be encrypted with hashing techniques before storing them in the database.
TIC-19 ACID Property Testing Database should follow all the ACID properties properly.
