Centralized Food Ordering Portal: Virginia Tech

Test Scope


Test Items

For the testing scope, we will be looking at the following components:
  • The GUI testing of web applications is required, and it will cover a range of GUI components including usability, functionality, accessibility, and more.
  • Through the Rest API, managers and vendors can speak with the system directly. Testing for both functionality and security should be done on APIs.
  • In order to ensure that data leaks and sql injection are not possible, security testing should also be done.

Functional Requirement

No. Requirement Scope Description
FR-1 The system must support and authenticate the login for dining hall managers, southgate manager and vendors.span> In scope The dining hall managers, southgate managers and vendors all should be able to create an account on the web application and log in to the system.
FR-2 The system must support the ability to check the availability of the products in the portal. In scope The users of the system must be able to check the availability of the products in the GUI.
FR-3 The system must support the ability to place orders for products and add or delete the number of items required. In scope The GUI must reflect the changes to quantities of items when altered by the user. The order should be placed successfully when no required fields are missing.
FR-4 The system must support the ability to request for items that are unavailable but necessary. In scope The users should be allowed to make the request for unavailable but necessary items.
FR-5 The system must support the ability to retrieve the orders and place them in the portal. In scope The system should retrieve the orders from the database successfully and display them to the relevant users in the GUI.
FR-6 The system must send confirmation of items once the orders have been accepted by the vendors. In scope The users should receive confirmation from the system when the vendor accepts the order.
FR-7 The system must support the ability to update the portal to reflect the availability of the items. In scope The relevant users should be able to update the item availability on the portal and the GUI should reflect this availability.
FR-8 The system must support the ability to give confirmation of delivery to the dining hall managers on the portal. Out of scope Once the order has been delivered, the dining hall managers should receive confirmation.


Non-Functional Requirement

No. Requirement Scope Description
NFR-1 The login credentials for dining hall managers, southgate managers and vendors shall be added to the system. In scope Once a user of a particular role is created, the credentials should be added to the database.
NFR-2 Background check of the vendors must be done before adding them as vendors to the system. Out of scope Testing the verifiability of background checks of the vendors is not in our scope.
NFR-3 The system should be scalable in order to support more and more vendors. In scope Adding more and more vendors should not cause the system to fail.
NFR-4 The system should be capable of supporting multiple dining hall managers logged in at the same time. In scope Multiple dining hall managers should be able to access the system without causing it to fail.
NFR-5 Each order request should be processed within 2 seconds. Out of scope Order requests should be processed within the time limit specified.
NFR-6 The site should load within 3 seconds when the number of simultaneous users is > 10. In scope The system should not fail when the number of simultaneous users is high.
NFR-7 The website should be supported on different browsers. In scope The web application should be able to run on different browsers.
NFR-8 The date format must be as follows: month/date/year. In scope The GUI should display dates in the specified format only.
NFR-9 There should be a dedicated printer so that the southgate manager can print the labels for the corresponding orders. Out of scope Verifying the printers for soutgate managers is not in the scope of our testing.


Exit Criteria

We have the following exit criteria:
  • All planned testing has been completed.
  • There shouldn't be any unresolved high severity or critical faults.
  • All of the high-risk areas have been carefully examined.
  • The code's coverage requirements ought to be met.
