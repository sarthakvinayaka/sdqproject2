Centralized Food Ordering Portal: Virginia Tech

Introduction


Objective

This test plan's objective is to create a benchmark for all future testing of this system. In addition to providing guidance on the overall testing strategy, including the types of testing to be conducted, various testing phases, exit criteria, and so on, this plan will serve as the basis for the minimal testing that must be carried out.

The baseline for all efforts connected to testing should be this document, which should be an addition to the system requirements and architectural changes.


System description

The system is designed to make the Southgate Dining hall operate more efficiently. It will automate the Virginia Tech Dining Halls' centralized meal ordering system. Southgate would serve as the portal's supreme administrator. The Manager would update the list of food items that needed to be ordered from the vendor on a regular basis. The Food Ordering Portal will have separate accounts for each dining hall. Each dining hall's manager is in charge of submitting requests through the portal for the food item and both the required and desired quantities. The system stops accepting requests for the day at 1 PM and automatically contacts the appropriate vendors to request the food items. The system notifies the dining halls of the anticipated items for the following day and the best vendor responds with the item that is currently in stock. The portal reflects the delivery to each dining hall once the items have been delivered.


Scope

The scope of this test plan is only for integration and validation testing. We have outlined the specifications, design elements, and test strategy in great detail. For each special test method, we have a separate phase in our test plans that includes a brief explanation of why building all test case skeletons for the test cases is necessary. All of our test cases are built on top of the traceability matrix.


System Architecture

Looking at the functional and nonfunctional requirements, it was evident that there would be 3 parts of the system:
  1. Front-end or UI of the application
  2. Back-end logic for providing the required functionality of the portal
  3. Database support to store all the data in a coherent way and to make data manipulation more efficient
The high level structure which would be used to design this web application will be Model-View-Template structure.
